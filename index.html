  • Standarization Info:

    The same standarization is used in every file and would not change.

    IN JS FILES: If some values has many duplicates in different tasks, their names has been changed with something that is sugar-like ID. In example, if I have variable "a" duplicated 3-4 times, in each task duplicated variables has other ID at the end of their name. Variable "a" in #Task 4 would stand for "aT4" (T4 is ID of Task 4).

    IN HTML FILES: Task IDs has always the same construction, they are started with name of parent div (for FreeCodeCamp ID shortcut would be "fcc". And so on: buttons attached to tasks has the same constructions but they are preceded by "run-". In example: for #Task 2, debug button ID would be "run-fcc-task-2", and span id inside debug box (from JS FILES -> debug scope) would be "fcc-task-2", which stands for Standard Task.

    Debug functions are based on debug buttons (with task ID, in ex. "run-fcc-task-2") and debug scopes (spans with task IDs, in ex. "fcc-task-2"), and their functionalities are all located inside JS files associated with tasks that are completed there. For better functionality, I left them all under the tasks instead of creating arrays or big lists, which simply would not be good idea while scopes still have to be defined, so Array would simply not change much instead of creating more code.

    In order to leave proper learning curve, constructions with old standards (ES5 etc.) hasn't been changed. That's why you may find some VARs inside JS files. (In example - there are some local/global scopes examples which need specific constructions)
